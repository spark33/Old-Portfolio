TEF Scheduler

I am a personnel manager for CMU's Computing Services. I created the TEF Scheduler to centralize all of TEF's software with enhanced user experience and functionality and replace the old WhenToWork system.

This application will be completed this Spring. We expect deployment to occur this Fall after several months of user testing.

User Interaction Design

Currently, my department uses WhenToWork to schedule employee shifts, Google Sheets to monitor hour logs and a custom LAMP app for employees to submit reports. Every shift change is made by an administrator who is informed by employees by e-mail, leading to flooded inboxes and many missed shift changes. No employee management system caters to student employees' short shifts and unpredictability in schedules.

TEF Scheduler was designed to maximize automation and flexibility in scheduling. It makes shifting easier for students while giving administrator a bird's eye view of all shifts and reports. In the future, it will also support data analytics and trend analysis.

The previous model keeps reports completely separate from the scheduling module. The Scheduler merges the reporting app with the scheduling app with shift-specific reports. This allows scalable development of data analysis tools on the same app.

User Profiles

Unlike WhenToWork's separate availability module, the Scheduler does not require the user to open external windows to input availability. All necessary user information can be viewed and edited on one page.

Scheduling Module

On WhenToWork, all functions were done on separate web pages. Shifts had to be manually subbed by deletion and addition by a manager. This module allows the user to view the schedule by building, pick up shifts, assign subs and submit reports using one simple jQuery calendar. This eliminated extraneous e-mail communication and ~20 clicks for every shift change.

View Reports

By displaying all reports on one view, the manager can simply scroll through reports instead of clicking on individual reports to read them.

Implementation

To prioritize flexibility in data storage and access, I decided to use a NoSQL database with AngularJS. Given the department's small size, denormalization was not a huge issue. The availability module and scheduler module use the FullCalendar framework. The application is styled using Materialize. The MongoDB database was hosted on MLab and the app was hosted on Heroku.


User Stories

As a(n)... I want to... So that...
Employee Find a sub for my shift I am not held accountable for an absence.
Employee Upload my availability and preferences I can get the shifts I want.
Employee View all empty shifts I can take an empty shift if I can.
Employee Pick up a temporary shift I can sub for someone else.
Employee Submit shift report I can report the work I've done.
Employee Report equipment damage or supply shortage I can notify staff.
Employee View my personal shifts I can figure out when I am working.
Employee Keep up with my performance I can figure out how I can be better.
Employee Give TEFman and staff feedback I can ask questions and help the department improve.
Manager Search up all reports i can follow up with my mentees if necessary.
Manager Check employees' availabilities so that I know who to give shifts to.
Manager Monitor reported hours so that I can make sure everyone is working all their shifts.
Manager Submit mentoring reports I can report progressing with mentoring mentees.
